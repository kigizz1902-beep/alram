# 최근뉴스 브리핑 기능 PRD

## 1. 배경 및 목표

### 배경
기존 알람 웹앱(간편 알람)은 설정한 시각에 알람 소리를 울리는 단순 기능만 제공한다. 사용자가 알람을 끈 직후 그날의 주요 뉴스를 바로 확인하고 들을 수 있다면, 아침 기상 루틴에서 별도로 뉴스 앱이나 라디오를 찾아보는 수고를 덜 수 있다. `최근뉴스_기술명세서.md`에 정리된 기술 검토(네이버 뉴스 검색 API + Supabase Edge Function 프록시 + Web Speech API)를 바탕으로, 알람 기능에 "뉴스 검색 + 읽어주기"를 통합한다.

### 목표
- 알람을 끈 시점을 트리거로 최신 뉴스 5건을 검색해 화면에 보여준다.
- 표시된 뉴스를 브라우저 내장 음성합성(TTS)으로 순차적으로 읽어준다.
- 뉴스 기능의 장애가 알람 본연의 기능(정해진 시각에 소리로 깨우는 것)을 절대 훼손하지 않도록 한다.
- 네이버 API 인증 정보를 안전하게 서버 측(Supabase Edge Function)에서만 관리한다.

### 비목표(이번 PRD에서 다루지 않는 것)
- 외부 고급 TTS(클로바더빙, ElevenLabs 등) 연동 — 향후 업그레이드 과제로 분리.

## 2. 사용자 스토리

1. **알람 끄기 → 뉴스 확인**
   사용자로서, 아침에 알람을 끄면 곧바로 오늘의 주요 뉴스 5개를 카드 형태로 보고 싶다. 그래야 침대에서 일어나자마자 세상 소식을 파악할 수 있다.

2. **뉴스 듣기**
   사용자로서, 화면을 보지 않고도(세안 중, 옷 갈아입는 중 등) 뉴스를 음성으로 듣고 싶다. 그래야 다른 아침 활동을 하면서 동시에 뉴스를 소비할 수 있다.

3. **재생 제어**
   사용자로서, 읽어주는 뉴스 중 관심 없는 항목은 건너뛰고 다음 뉴스로 넘어가거나, 필요하면 즉시 멈추고 싶다. 그래야 내가 원하는 정보만 골라 들을 수 있다.

4. **네트워크/서비스 문제 시에도 안심**
   사용자로서, 뉴스 검색이 실패하거나 느리더라도 알람이 울리는 것 자체와 알람을 끄는 동작에는 전혀 지장이 없기를 바란다. 그래야 뉴스 기능 하나 때문에 지각하는 일이 없다.

5. **에러 상황 인지**
   사용자로서, 뉴스를 불러오지 못했을 때 화면이 멈추거나 아무 반응이 없는 대신, "뉴스를 불러올 수 없습니다" 같은 명확한 안내를 보고 싶다.

## 3. 기능 요구사항

### 3.1 뉴스 검색
- FR-1. 알람이 울린 뒤 사용자가 알람을 끄는 클릭/버튼 조작을 하면, 이를 트리거로 최신 뉴스 5건 검색을 시작한다.
- FR-2. 뉴스 검색은 React 프론트엔드 → Supabase Edge Function(프록시) → 네이버 뉴스 검색 API 순서로 요청/응답이 전달된다.
- FR-3. Edge Function은 네이버 API 원본 응답에서 제목, 요약, 링크, 출처, 발행일 등 필요한 필드만 정제하여 프론트엔드에 반환한다.
- FR-4. 프론트엔드는 응답을 받아 뉴스 카드 UI(제목, 요약, 출처/발행일, 원문 링크)로 목록을 렌더링한다.
- FR-5. 뉴스 응답 대기 중에는 로딩 상태를 표시한다(약 1~2초 지연 가능성을 고려).

### 3.2 뉴스 읽어주기(TTS)
- FR-6. 뉴스 카드가 표시되면 첫 번째 뉴스부터 `speechSynthesis`를 이용해 자동으로 읽기 시작한다(사용자가 알람을 끈 클릭을 사용자 상호작용으로 활용하여 브라우저 자동재생 제약을 준수).
- FR-7. "다음 뉴스", "정지"(및 필요 시 "일시정지/재생") 컨트롤 버튼을 제공한다.
- FR-8. 현재 읽고 있는 뉴스 항목을 화면에서 시각적으로 강조 표시한다.

### 3.3 알람 연동
- FR-9. 알람 소리 재생 로직과 뉴스 기능 로직은 독립적으로 동작해야 하며, 알람 소리는 뉴스 기능의 성공/실패와 무관하게 항상 먼저, 정상적으로 울린다.
- FR-10. 사용자가 알람을 끄는 즉시 알람 소리는 멈추고, 그 직후(비동기로) 뉴스 검색이 시작된다. 뉴스 검색/재생이 알람을 끄는 동작 자체를 지연시키지 않는다.

### 3.4 에러 처리
- FR-11. 뉴스 검색 실패(네트워크 오류, API 오류, 타임아웃 등) 시 사용자에게 "뉴스를 불러오지 못했습니다" 등 명확한 안내 메시지를 표시하고, TTS 및 컨트롤 버튼은 비활성화하거나 숨긴다.
- FR-12. 사용자 브라우저가 `speechSynthesis`를 지원하지 않거나 한국어 음성을 지원하지 않는 경우, TTS 없이 텍스트(카드)만 정상적으로 표시한다.

## 4. 비기능 요구사항 (보안 최우선)

### 4.1 보안 (필수 — 최우선 순위)
- NFR-1. **네이버 API 키(Client ID/Secret)는 프론트엔드에 절대 노출하지 않는다.** Client ID/Secret은 Supabase Edge Function의 환경변수(Secrets)에만 저장하며, 클라이언트(브라우저) 코드, 번들, 네트워크 응답, 로그 등 어디에도 포함되어서는 안 된다.
- NFR-2. 네이버 뉴스 검색 API 호출은 반드시 Supabase Edge Function을 경유하며, 브라우저에서 네이버 API 엔드포인트를 직접 호출하는 코드는 존재해서는 안 된다.
- NFR-3. Edge Function은 자신이 정제한 응답 필드(제목/요약/링크/출처/발행일 등) 외의 불필요한 원본 데이터나 인증 관련 정보를 클라이언트로 전달하지 않는다.
- NFR-4. Edge Function 호출 시 CORS 설정은 필요한 출처(origin)로 제한하는 것을 권장한다.

### 4.2 데이터 처리
- NFR-5. **뉴스 데이터는 DB에 저장하지 않고 매 요청마다 실시간 조회한다.** 검색 결과 캐싱이나 로그 저장(기술명세서 2절에서 언급된 "고려 가능" 옵션)은 이번 범위에서 채택하지 않는다.
- NFR-6. 사용자의 개인 식별 정보를 뉴스 검색 요청에 포함하거나 별도로 수집하지 않는다.

### 4.3 신뢰성 / 가용성
- NFR-7. **뉴스 브리핑 기능이 실패해도 알람 소리 자체는 반드시 울려야 한다.** 알람 트리거·소리 재생 로직은 뉴스 검색/TTS 로직과 강하게 분리(디커플링)되어, 뉴스 관련 코드의 예외나 응답 지연이 알람 소리 재생에 영향을 주지 않아야 한다.
- NFR-8. 뉴스 API 호출에는 타임아웃을 설정하여 무한 대기 상태를 방지한다.

### 4.4 성능
- NFR-9. 뉴스 검색 응답은 평균 2초 이내를 목표로 하며, 지연 시 로딩 상태를 명확히 표시한다.

### 4.5 호환성
- NFR-10. 주요 데스크톱 브라우저(Chrome 기준)에서 정상 동작을 보장하며, 모바일/기타 브라우저의 음성 품질 차이는 감수하되 기능 자체(카드 표시)는 항상 동작해야 한다.

## 5. 범위 제외 (Out of Scope)
- 외부 고급 TTS 서비스(네이버 클로바더빙, ElevenLabs 등) 연동
- 뉴스 검색 결과의 DB 캐싱/로그 저장
- 완전한 백그라운드(탭이 닫히거나 백그라운드 상태일 때) 알람 및 뉴스 재생 — 브라우저 탭이 열려 있어야 함을 전제로 함
- 사용자별 뉴스 카테고리/키워드 개인화 설정
- 다국어 뉴스 검색 및 다국어 TTS
- 완전 자동(사용자 클릭 없이) 알람 종료 + 뉴스 재생 — 브라우저 자동재생 정책상 사용자 상호작용을 트리거로 사용

## 6. Open Questions
- Supabase 프로젝트를 신규로 구성해야 하는지, 기존에 사용 중인 Supabase 프로젝트가 있는지?
- 네이버 개발자센터 애플리케이션(Client ID/Secret) 발급 주체는 누구이며, 발급 완료 시점은 언제인가?
- 뉴스 검색 실패 시 재시도(retry) 로직이 필요한가, 아니면 즉시 에러 메시지만 표시하면 되는가?
- "다음 뉴스" 버튼 클릭 시 현재 읽던 문장을 즉시 중단하고 다음으로 넘어가는지, 아니면 현재 문장은 끝까지 읽는지?
- Edge Function 요청에 대한 rate limiting(과도한 호출 방지)이 필요한가?
- 알람이 여러 개 겹쳐 울리는 경우 뉴스 기능은 어떻게 동작해야 하는가(예: 마지막에 끈 알람 기준으로만 동작)?

## 7. 성공 기준
- 알람을 끈 뒤 정상적인 네트워크 환경에서 3초 이내에 뉴스 카드 5건이 표시된다.
- 카드 표시와 동시에(또는 즉시 이어서) 첫 번째 뉴스부터 TTS 음성 읽기가 자동으로 시작된다.
- "다음 뉴스", "정지" 컨트롤이 의도한 대로 동작한다.
- 네이버 API Client ID/Secret이 프론트엔드 번들, 브라우저 네트워크 탭, 콘솔 로그 등 어디에서도 발견되지 않는다(코드 리뷰 및 브라우저 개발자도구 점검으로 검증).
- 뉴스 데이터가 어떤 DB 테이블에도 저장되지 않음을 확인한다(Supabase Database 스키마/로그 점검).
- 네이버 API 또는 Edge Function을 인위적으로 실패시키는 테스트에서, 알람 소리는 정상적으로 울리고 알람을 끄는 동작도 정상 동작함을 확인한다.
- 뉴스 검색 실패 시 사용자에게 에러 안내 메시지가 표시되고, 앱이 멈추거나 크래시하지 않는다.
