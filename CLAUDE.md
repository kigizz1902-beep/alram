이 프로젝트에서 작업할 때 항상 지켜야 할 공통 규칙입니다. 기능별 세부 규칙은 각 기능 폴더의 CLAUDE.md를 따로 참고하세요.

## 🚨 최우선 보안 규칙

**네이버 API 키(Client ID/Secret)는 어떤 경우에도 프론트엔드 코드에 절대 넣지 않는다.**

- Client ID/Secret은 반드시 서버 측(Supabase Edge Function 등)의 환경변수(Secrets)에만 저장한다.
- 브라우저 번들, React 컴포넌트, 클라이언트 JS 코드, `.env`가 아닌 커밋 파일, 콘솔 로그, 클라이언트에서 확인 가능한 네트워크 응답 등 어디에도 API 키가 노출되어서는 안 된다.
- 외부 API(네이버 등) 호출이 필요한 경우, 클라이언트가 외부 API를 직접 호출하지 않고 반드시 서버/프록시(Edge Function)를 경유하도록 구현한다.
- 코드를 작성하거나 수정할 때 이 규칙 위반 여부를 항상 스스로 점검한다.

## 보안 및 데이터 처리 원칙

- 인증 정보(API 키, 시크릿 등)는 절대 하드코딩하지 않고 서버 측 환경변수로만 관리한다.
- 외부 API를 경유하는 프록시(Edge Function 등)는 클라이언트에 필요한 필드만 정제해서 반환하고, 원본 응답이나 인증 관련 정보를 그대로 전달하지 않는다.
- 조회성 데이터(예: 뉴스 등 실시간성이 중요한 데이터)는 DB에 저장하지 않고 매 요청마다 실시간으로 조회하는 것을 기본 원칙으로 한다. DB 캐싱/로그 저장이 필요하다고 판단되면 반드시 사용자에게 먼저 확인한다.
- 사용자의 개인 식별 정보를 불필요하게 수집하거나 요청에 포함하지 않는다.

## 실패 허용(Fault Tolerance) 원칙

- 부가 기능(예: 뉴스 브리핑 등 알람 이후에 이어지는 기능)의 실패가 핵심 기능(예: 정해진 시각에 알람 소리가 울리는 것)에 절대 영향을 주어서는 안 된다.
- **뉴스 브리핑 기능이 실패해도 알람 소리 자체는 반드시 울려야 한다.** 핵심 기능과 부가 기능은 코드 상에서 강하게 분리(디커플링)하여, 부가 기능의 예외·지연·오류가 핵심 기능의 실행을 막거나 지연시키지 않도록 구현한다.
- 외부 API 호출에는 타임아웃을 설정하여 무한 대기를 방지하고, 실패 시 사용자에게 명확한 안내 메시지를 표시한다(화면이 멈추거나 아무 반응 없는 상태를 만들지 않는다).

## 작업 순서 원칙

새로운 기능을 구현할 때는 다음 순서를 따른다.

1. 먼저 가짜(mock) 데이터로 화면(UI)을 완성한다.
2. 그다음 실제 백엔드/외부 API 연동을 붙인다.
3. 예외 처리(에러 메시지, 타임아웃, 폴백 등)는 가장 마지막에 추가한다.

이 순서를 임의로 바꾸거나 건너뛰지 않는다.

## 미확정 사항(Open Questions) 처리 원칙

각 기능의 PRD(`prd.md`)에 정리된 Open Questions(예: 뉴스 재생 컨트롤의 세부 동작 방식, 재시도/rate limiting 정책 등)는 아직 확정되지 않은 사항이다. 이런 항목은 임의로 결정하거나 추측으로 구현하지 말고, 구현이 필요한 시점에 반드시 사용자에게 먼저 확인한다.
